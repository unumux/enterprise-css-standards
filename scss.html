Unum Enterprise CSS/SCSS Standards

SCSS Formatting

SCSS Standard Project Folder Structure

  • The standard project structure sould mirror this standard project structure located in github.
    1. Only one .css file in the root (minified)
    2. Only one .scss file in the root
    3. Two Sibling folders: 'Components' and 'Theme'

SCSS Standard Partial Structure

  • All sass partials should be prefixed with an underscore and use the .scss file extension: 
          _banner.scss

SCSS Variables

  1. Variable names should not be how something looks, but rather what the something is responsible for: 
         // This is ok:
   $ocean: #00a3bb;

   // This is better:
   $primary-color: #00a3bb;

   // This is even BETTER better:
   $component-banner-background: $primary-color;
  2. Use imported or already existing branding or theme colors as bases for your variables, abstracted out and ensure you add 'default' at the end of any variables you create/use that you might need to easily override in a later partial: 
        // Variable from the 'descriptive-colors.scss' theme partial:
    $color-ocean: #3baecc;

    // Using the descriptive color in the theme palate:
    $color-text-primary: $color-ocean !default;

    // Using the theme palate color in a component variable:
    $component-text-heading: $color-text-primary !default;
  3. Don't name variables based on context: 
        // Wrong
    $background-color: $color-ocean !default; 

    // Right
    $color-text-primary: $color-ocean !default;
  4. Create variables when the value is repeated more than once, or the value is likely to be updated more than once, or all occurrences of the value are tied to the variable.
  5. The !global flag should only be used when overriding a global var from a local scope.
  6. Variables across the whole scss codebase should be placed in their own file - this needs team discussion
  7. Set variables for breakpoints or create mixins that you can update easily - team discussion for actual implementation

SCSS Mixins

  1. All .SCSS component-named partials should keep the base styles in a base mixin that is contained at the top of the document: 
        // This is scss in an individual component root file - for ex. 'banner.scss'
    @mixin base() {
        text-align: center;
    }
     
   .banner {
        @include base;
    }
  2. Try to use mixins whenever possible for any groups of properties that are repeated and located together.
  3. Use mixins when things require parameters.
  4. Use mixins to change size.

SCSS Selector Construction (Property:Value organization)

  1. @extends;
    • Do not use in day to day code unless there is a need
  2. @includes;
    • Should be at the top of the selector
  3. declaration list (property: name;);
  4. :pseudo-states/elements;
  5. media queries{};
  6. { nested elements {}};
  7. { nested classes {}};
  8. @imports;
    • Should be at the bottom of the selector
  9. repeat the above hierarchy within nested elements/classes.
  10. For a great example of this setup in context, view our component scss setup in our component framework.
