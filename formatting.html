Unum Enterprise CSS/SCSS Standards

CSS Rule Formatting

Spacing

  1. Use tab indention
  2. Tabs should equal 4 spaces
  3. Use a space between the opening brace and the class name: 
        // Wrong
    .example-class{...}

    // Right
    .example-class {...}
  4. Vendor prefixes should be aligned: 
        .example-class {
        -webkit-border-radius: 3px;
           -moz-border-radius: 3px;
                border-radius: 3px;
      }
  5. Combinators (and any calc or SCSS computations) should have a space before and after: 
        .example-class > .example-class {
        height: calc(50% - 1rem);
    }
  6. Ensure all calculations are wrapped in parentheses: 
        .example-class {
        width: calc(100vw / 2);
    }
  7. Closing selector braces should be in line with the selector name and on their own line: 
        .example-class {
        color: #fff;
    } // This brace should be in line with the namespace

Measurements

  1. Use percentages when layout components stay relational to each other (like a component that always takes up 75% of the screen and a second one that always takes up 25%)
  2. Use pixel(px) units for when measurement should not change based on user-set font size or browser zooming.
  3. Use rem units for font sizes.
  4. Use up to 10 decimal places in em units to ensure accuracy - if you choose to do so, it will eliminate some of the browser-specific calculation fluctuations based on resizing and percent widths etc.
  5. Use units when measuring dimensions (margins, borders, padding etc).

Property Declaration Order

  1. for a full list of all SCSS properties and their orders, read the SCSS properties.
  2. Property declaration should be alphabetical
  3. Property declarations should not be written in shorthand. Common overused shorthand properties include:
    • padding
    • margin
    • font
    • background
    • border
    • border-radius
        // Wrong
    .example-class {
        padding: 0 0 1rem 0;
    }
    
    // Right
    .example-class {
        padding-bottom: 0;
        padding-left: 0;
        padding-right: 0;
        padding-top: 1rem;
    }
  4. Only include properties being used in rule declarations. For example, if there are no styles necessary for 'border-top', then do not declare it in the rule.
  5. Remove empty rulesets
  6. Selectors and declarations should be written on single lines: 
        .a,
    .h4,
    .p {
        color: blue;
        font-size: 1.25rem;
    }
  7. Rulesets should have an empty line between them: 
        .example-class {
        property-name: value;
    }
  
    .example-class {
        property-name: value;
    }
  8. Use all lowercase when creating a class or variable, or when referencing a div or element: 
        .example-class {...}

    $component-heading-text: #ccc;
  9. Use dashes over camelCasing in class names. Underscores are used as part of the BEM methodology. 
        // This is a good -if verbose- example:
    .example-class__heading--postive {...} 

    // This is terrible don't do this:
    .thisIsBad_dontDoThis-BadStuff {...}
  10. Use double quotes for properties: 
        .example-class__heading--positive {
        font-family: "Open Sans", helvetica, sans-serif;
        background-image: url("/images/logo.jpg");
    }
  11. All property-value pairs should be on a new line, indented once and have an ending semi-colon: 
        .example-class {
        color: #fff;
        font-weight: 300;
        text-decoration: underline;
    }
  12. Any comma-separated property values should have a space after each comma.
  13. Do not include spaces after commas within () values (like rgb(), rgba() etc).

Colors

  1. Hexadecimal codes should be in lowercase
  2. Use shorthand hex notation where available: 
        .example-class {
        //regular 6 char hex notation
        color: #ffffff; 
    }
  
    // Right
    .example-class {
        //shortened 3 char hex notation
        color: #fff; 
    }
  3. Do not use named colors: 
        // Wrong
    .example-class {
        color: blue;   
    }
  4. For examples of color variables, see the section in SCSS Color Variables

Comments

  1. Well-commented code is important, take time to properly describe what the code does, limitations, construction etc.
  2. Try to use line comments (//) over block comments (/* ... */)
  3. Put comments on their own line above the class or declaration in question, and avoid end-of-line comments
  4. Write detailed comments for code that isn't self-documenting:
    • Documenting uses of z-index
    • Compatibility or browser-specific hacks
  5. Ensure comments are user-friendly (can easily be understood by other devs)
  6. highlight purpose or context of item being explained in comment: 
        // Wrong
    // Class Name
    .example-class {...}

    // Right
    // Wrapping element for .example-class__heading
    .example-class {...}
  7. Length of code or comment lines should be 80 characters or less: 
        // Dang this is a really long comment, it really had to detail in depth
    // everything regarding what happened in this particular segment of css,
    // it should be easy to read both vertically and horizontally 
    // without having to scroll in order to read it all

Fonts

  1. Use number values for font weights, not words.
  2. Use rem units for font sizes with a px fallback. 18f example with mixin.
  3. Set the HTML element font size to a default size (10px) so .1rem = 1px for easier to maintain calculation later on.

Media Queries

  1. Try to specify in the rule what your MQ is for (screen, tv, handheld etc) so it will not be applied to ALL stylesheets: 
        .example-class {...}
    
    @media screen (min-width: 680px) {
       .example-class {...}
    }
  2. Use relative units for MQ's, avoid using pixel-based units.
  3. Place media queries as close to relevant rulesets as possible. Do not bundle them in a separate stylesheet(print is an exception) or bunch them at the bottom of the doc: 
        .example-class {...}
    
    @media screen (min-width: 680px) {
       .example-class {...}
    }
  4. Using SCSS you should nest your MQ in the relevant class: 
        .example-class {
        background-color: lighten($primary-component-background, 10%);
    
        @media screen (min-width: 680px) {
            background-color: $primary-component-background;
        } 
    }
    Note: this will compile to the same structure as #3 example above.
  5. Use variables to set your MQs throughout so they are easy to manage and update
  6. Avoid(if possible) using an arbitrary number of media queries or breakpoints

Values

  1. Avoid using arbitrary numeric values for properties: 
        // Wrong
    .example-class {
        margin-top: .37rem;
    }

    // Right
    .example-class {
        margin-top: $computed-shared-value;    
    }
    // Where that value is used across multiple components
    // for consistent appearance
  2. Values less than 1 should have a leading zero (0.5rem etc)
  3. Be consistent with the units used throughout your document
  4. Line-height value should be unitless: 
        .example-class {
        line-height: 1;
    }
  5. Use a value of '0' instead of 'none'' to specify an element has no border or padding or margin etc: 
        .example-class {
        border: 0;
    }
  6. Avoid specifying units for zero values: 
         // Wrong
    .example-class {
        border: 0px;
    }

    // Right
   .example-class {
        border: 0;
    }

HTML, Attribute, Class and ID Selectors

HTML Selectors

  1. When styling HTML elements, use only a supplied theme or class (see the Willow Component Library for examples of styling HTML elements): 
        .component-context {
        h1 {
            font-weight: 700;
        }
    }
  2. Do not use tag selectors unnecessarily: 
        // Wrong
    p.example-class {
        font-weight: 700;
    }

    // Right
    .example-class {
        font-weight: 700;
    }

Attribute Selectors

  1. Use HTML and attribute selectors only when necessary.
  2. Attribute values in selectors should be surrounded by double quotes: 
        // Attribute selector:
    [type="input"] {...}

    // Element selector
    h1 {...}

Classes

  1. Try to use only classes to style the project
  2. Avoid using more than one class per element wherever possible

IDs

  1. Do not use IDs as selectors

Specificity

  1. Use class name(s) to apply styles. Avoid using elements for as much as possible except base theme styles.
    • This decreases specificity issues, leaky css, and aids in future-proofing code.
  2. Scope classes to closest parent only when necessary.

Nesting

  1. Try to avoid using nested selectors.
    • If you have no choice, apply styles to an element by using nested rulesets no more than three selectors deep.
    • BEM principles will help restrict and enforce this convention for most use cases

JavaScript Hooks

  1. Sometimes you will need to use JS in order to hook into a module. In that case, a separate class, prefixed with 'js' is recommended: 
        class="primary-feature--headline js-toggle"
  2. Do not use a 'js-' class to attach styles to an element.
